# Roadmap

每个 Phase 完成后，Agent 必须更新 `docs/CURRENT_STATE.md` 和 `docs/CHANGELOG_AGENT.md`，再运行 `python3 scripts/agent_changelog_archive.py` 刷新 `docs/RECENT_HISTORY.md`。

## Phase 0：项目规划文档

- 目标：建立 Agent 可长期维护的文档体系。
- 后端任务：定义技术栈和模块边界。
- 前端任务：定义目录和交互方向。
- 数据库任务：设计核心表。
- AI 任务：定义 provider 策略。
- 验收标准：必需文档完整，Backlog 可执行，状态可接手。

## Phase 1：项目骨架

- 目标：创建前后端项目、基础目录、Docker Compose 和健康检查。
- 后端任务：Spring Boot 基础服务、健康检查、基础安全配置。
- 前端任务：Vue 3 基础应用、路由、状态管理、首页。
- 数据库任务：MySQL Compose、Flyway 基础配置。
- AI 任务：保留 provider 设计入口，不接真实服务。
- 验收标准：后端测试通过，前端构建通过，README 可启动。

## Phase 2：用户与鉴权

- 目标：实现注册、登录、JWT 鉴权、当前用户信息。
- 后端任务：用户实体、密码加密、JWT、鉴权过滤器、接口。
- 前端任务：登录注册页、Auth Store、路由守卫。
- 数据库任务：`user` 表和迁移。
- AI 任务：无。
- 验收标准：受保护接口需要 token，前端可登录并保持会话。

## Phase 3：生活空间 / 家庭成员

- 目标：支持个人空间、家庭空间和成员权限。
- 后端任务：空间、成员、角色、权限校验。
- 前端任务：空间切换、成员管理基础界面。
- 数据库任务：`household`、`household_member`。
- AI 任务：后续基于空间隔离 AI 上下文。
- 验收标准：用户只能访问自己所属空间的数据。

## Phase 4：智能记账

- 目标：收入、支出、分类、预算和基础分析。
- 后端任务：交易记录 CRUD、分类管理、预算接口。
- 前端任务：记账表单、列表、分类筛选、预算视图。
- 数据库任务：`transaction_record`、`transaction_category`、`budget`。
- AI 任务：为自然语言记账准备结构化 DTO。
- 验收标准：空间内可管理记账数据并统计月度总额。

## Phase 5：购物清单

- 目标：管理购物清单、清单项、状态和预算估算。
- 后端任务：购物清单 CRUD、清单项状态变更。
- 前端任务：清单列表、编辑、采购状态。
- 数据库任务：`shopping_list`、`shopping_item`。
- AI 任务：后续支持 AI 生成购物清单。
- 验收标准：用户可在空间内维护购物清单。

## Phase 6：家庭库存

- 目标：管理食品、日用品、常备物品和提醒。
- 后端任务：库存 CRUD、临期和缺货查询。
- 前端任务：库存列表、分类、位置、提醒视图。
- 数据库任务：`inventory_item`。
- AI 任务：为菜谱推荐提供库存输入。
- 验收标准：用户可查看库存状态和提醒。

## Phase 7：生活待办

- 目标：普通待办、重复任务、提醒和完成率统计。
- 后端任务：任务 CRUD、重复规则、完成状态。
- 前端任务：待办列表、日历或分组视图。
- 数据库任务：`todo_task`。
- AI 任务：自然语言创建待办。
- 验收标准：用户可创建、完成、统计任务。

## Phase 8：饮食计划

- 目标：菜谱和一周菜单。
- 后端任务：菜谱管理、菜单计划、库存关联。
- 前端任务：菜谱页、周计划页。
- 数据库任务：`recipe` 和计划表。
- AI 任务：根据库存推荐菜谱，提醒为生活建议非医疗建议。
- 验收标准：能基于库存生成温和菜谱灵感。

## Phase 9：AI 生活助手

- 目标：统一 AI provider、自然语言录入和分析。
- 后端任务：AiProvider 接口、mock provider、调用日志。
- 前端任务：AI 输入框、结果确认界面。
- 数据库任务：`ai_call_log`。
- AI 任务：自然语言记账、待办、购物清单、总结。
- 验收标准：AI 输出结构化，用户确认后才写入业务数据。

## Phase 10：数据统计与生活报告

- 目标：消费、购物、库存、待办和月度报告。
- 后端任务：统计聚合接口。
- 前端任务：仪表盘和图表。
- 数据库任务：优化索引。
- AI 任务：月度生活总结和优化建议。
- 验收标准：图表数据来自真实接口。

## Phase 11：票据与文件管理

- 目标：发票、收据、保修卡、合同、说明书和证件索引。
- 后端任务：文件记录和元数据管理。
- 前端任务：文件列表、筛选、详情。
- 数据库任务：`document_record`。
- AI 任务：文本信息提取，OCR 后期再做。
- 验收标准：用户可手动录入并检索文件记录。

## Phase 12：部署、测试、演示与优化

- 目标：完善 CI、部署说明、演示数据和端到端质量。
- 后端任务：配置生产 profile、接口稳定性。
- 前端任务：交互优化、错误态、空态。
- 数据库任务：迁移审计和备份策略文档。
- AI 任务：降级和限流策略。
- 验收标准：可稳定演示，测试和构建自动化。

## Phase 13：前端测试体系

- 目标：让前端 store、API helper 和关键交互具备可持续测试基础。
- 后端任务：无。
- 前端任务：接入 Vitest，覆盖认证、空间、HTTP 拦截器和主要 API 模块。
- 数据库任务：无。
- AI 任务：AI API 客户端测试覆盖请求路径和响应处理。
- 验收标准：`npm test` 和 `npm run build` 通过，新增前端 API 或 store 时有测试入口可复用。

## Phase 14：真实 AI Provider 准备

- 目标：在不提交真实密钥的前提下，为 OpenAI-compatible provider 做好配置、条件注入和安全边界。
- 后端任务：Provider 配置骨架、OpenAI provider 代码骨架、Mock 回退和测试。
- 前端任务：保持 AI 功能走统一 API，不暴露密钥。
- 数据库任务：后续可补 AI 调用日志和费用统计。
- AI 任务：真实 provider 输出必须保持结构化，失败时可回退或提示用户确认。
- 验收标准：`provider=mock` 行为不变；`provider=openai` 配置明确；无真实 API Key 进入仓库。

## Phase 15：可视化体验增强

- 目标：把统计和生活数据以图表方式呈现，提升首页和分析页面的信息密度。
- 后端任务：补齐统计接口，保证图表数据来自真实接口。
- 前端任务：ECharts 通用组件、首页统计图表和响应式展示。
- 数据库任务：按查询热点补索引。
- AI 任务：后续月报和建议可引用统计图表数据。
- 验收标准：图表数据真实、空数据不误导、构建和前端测试通过。

## Phase 16：用户体验增强与功能完善

- 目标：补齐 PRD 中 MVP 以外的功能点，增强用户体验和系统健壮性。
- 后端任务：用户个人资料编辑 API、库存临期/缺货提醒、待办完成率统计、购物预算字段。
- 前端任务：个人设置页面、首页提醒列表、预算编辑 UI。
- 数据库任务：购物清单预算字段迁移。
- AI 任务：后续可基于库存提醒和完成率数据生成生活建议。
- 验收标准：用户可编辑个人资料；首页展示库存提醒和待办完成率；购物清单支持预算金额。

## Phase 17：饮食计划

- 目标：实现一周饮食计划管理，支持按日期和餐次关联菜谱。
- 后端任务：meal_plan 表和实体、MealPlanService CRUD、按日期范围查询。
- 前端任务：周历视图（7 天 × 4 餐次网格）、周导航、今日高亮、菜谱选择弹窗。
- 数据库任务：V9 创建 meal_plan 表。
- AI 任务：后续可根据库存和饮食计划生成购物清单建议。
- 验收标准：用户可在周历视图中规划每周饮食；支持按日期范围查询；导航栏显示"饮食计划"入口。

## Phase 18：智能饮食推荐

- 目标：AI 根据当前库存物品匹配菜谱食材，推荐可用现有食材制作的菜谱。
- 后端任务：AiProvider 接口新增 `recommendRecipes`、MockAiProvider 关键词匹配评分、OpenAiProvider 委托 Mock、AiService 和 AiController 新增端点。
- 前端任务：MealPlanView 新增"AI 菜谱推荐"面板（匹配百分比、已匹配/缺失食材、推荐理由、点击填入饮食计划）。
- 数据库任务：无（使用已有 recipe 和 inventory 表）。
- AI 任务：mock provider 使用确定性匹配算法，不接真实 AI；后续可替换为真实 LLM 推荐。
- 验收标准：`GET /api/ai/spaces/{spaceId}/recommend-recipes` 返回推荐列表；前端推荐面板可展示和交互；后端单元测试通过。

## Phase 19：饮食采购闭环

- 目标：根据一周饮食计划和当前库存生成购物清单草稿，打通“库存 → 菜谱 → 饮食计划 → 采购清单”的生活数据闭环。
- 后端任务：AiProvider 接口新增 `draftShoppingListFromMealPlan`；MockAiProvider 解析菜谱食材并计算库存缺口；AiService 按日期范围查询 meal_plan、recipe、inventory；AiController 新增草稿端点。
- 前端任务：MealPlanView 新增"生成采购清单"按钮、草稿面板和确认创建购物清单流程，复用现有 shopping API 写入清单和清单项。
- 数据库任务：无（复用 meal_plan、recipe、inventory_item、shopping_list、shopping_item）。
- AI 任务：mock provider 使用确定性缺口计算；真实 provider 当前仍委托本地算法，不自动下单，不绕过用户确认。
- 验收标准：`GET /api/ai/spaces/{spaceId}/meal-plan-shopping-draft?startDate=&endDate=` 返回购物清单草稿；前端可展示草稿并在用户确认后创建购物清单；后端和前端测试通过。
